What Success Looks Like

Q2 Goals

Make the Transformation OS good enough that a client transformation lead considers it their system — not a consultant's tool they occasionally peek at.

1 A client can log in, see the current state of their transformation, and understand what needs attention — without needing a call with Tatoma.
2 The quarterly refresh workflow is guided enough that a transformation lead can prepare for a check-in session in under 30 minutes.
3 The platform surfaces signals (progress, blockers, upcoming activities) that justify its existence between consulting touchpoints.
4 Transformation leads can scope new use cases and request sprints independently from the platform.
5 The experience feels like a product, not a consulting deliverable viewer.

Guided Onboarding

From "first login" to "I understand my transformation" — a self-guided path for transformation leads and a setup wizard for consultants.

  • Welcome flow — guided walkthrough of the OS, what it does, what actions to take
  • Playbook setup wizard — structured consultant flow for new client setup
  • Sprint introduction — explain Spikes, Sprints, Specials during onboarding
  • Persistent progress checklist that disappears once key actions are taken

2

Living Playbook + Adoption Tools

Add strategic context, complete the placeholder tabs, add health signals — make the playbook a product, not a card viewer.

  • Playbook Overview page — rich-text transformation story as landing view
  • Phase summaries with per-client context at top of Align / Activate / Adopt
  • Winning moves as first-class navigation items with own pages
  • Timeline view, Stream view, Relations view (React Flow graph)
  • Client-side commenting on activities
  • Progress dashboard — health signals, scorecard, overdue/stale flags

3

Execution Use Case Generator

Let transformation leads capture and evaluate AI use cases independently — no more messy lists brought to quarterly sessions.

  • "New Use Case" guided form — department, process, pain point, impact, complexity
  • AI-assisted scoping via Groq — drafts preliminary brief from input
  • Use case backlog with prioritization matrix (impact vs. effort quadrant)
  • Consultants review and promote to activities or sprint candidates

4

Guided Refresh

The core value-between-sessions mechanism. "Before you get back, we do the platform-led refresh."

  • Refresh prompt — banner/card 2-4 weeks before quarterly session
  • Step-by-step questionnaire (sentiment, winning move status, blockers)
  • Document upload with AI context extraction via research pipeline
  • AI-generated pre-session briefing from responses + activity data
  • Automated reminder emails at 4 and 2 weeks
quarterly_refreshes refresh_documents
5

Quarterly Check-in Support

Platform support for the session itself — from consultant prep to capturing outcomes back into the system.

  • Pre-session briefing page — auto-generated from refresh + activity health + use cases
  • Session outcome capture — decisions, modified activities, next quarter priorities
  • Quarter-over-quarter comparison — completion rates, sentiment trend
quarterly_sessions
6

Transformation Lead MiniMBA

The TL role is demanding and often new. Give them a structured growth path — stakeholder management, change communication, initiative prioritization.

  • Curated learning path — short modules authored by Tatoma consultants
  • Contextual nudges — surface relevant content at the right moment
  • Progress tracking visible to both TL and consultant
  • Future: community of practice connecting TLs across clients
learning_modules learning_progress
7

Sprint Workflow

Spikes, Sprints, and Specials — scoped, requested, and tracked from within the platform. The bridge between strategy and execution.

  • "Request a Sprint" from activity cards or use cases
  • Structured brief template pre-populated from context
  • AI-assisted brief generation via Groq
  • Three types: Spike (~2d) · Sprint (~1w) · Special (deep-dive)
  • Sprint tracking: briefed → scheduled → in progress → delivered → reviewed
  • Outcomes captured back into the playbook

How to Build It

Implementation Phases

Four sequential phases over 12 weeks. Features within each phase can be parallelized.

A
Weeks 1–3

Foundation

Deploy the Living Playbook, add the narrative layer, and get the first-login experience right.

  • Complete Living Playbook deployment — merge, migrate, verify
  • Build guided onboarding flow — welcome walkthrough + progress checklist
  • Playbook narrative layer — overview page, phase summaries, winning moves pages
  • Timeline view using existing activity_log data
  • Client-side commenting on activities
  • Progress indicators — completion percentages on winning moves and phases
Exit criteria: A new client can log in for the first time and understand their transformation through a guided experience. Existing clients see their story, not just cards.
B
Weeks 4–6

Refresh & Check-in

Build the workflow that makes the platform valuable between sessions.

  • Guided refresh questionnaire — step-by-step pre-session flow
  • Document upload with AI context extraction
  • AI-generated refresh summary / pre-session briefing
  • Pre-session briefing page for consultants
  • Session outcome capture in Workbench
  • Refresh reminder emails via Resend
Exit criteria: TL receives reminder, completes refresh in ~20 min, consultant has briefing ready, session outcomes feed back into the platform.
C
Weeks 7–9

Intelligence & Nudges

Make the platform proactive and add the use case generator.

  • Execution use case generator — guided capture + AI scoping
  • Use case backlog with prioritization matrix
  • Activity health signals — overdue and stale detection
  • Transformation scorecard summary widget
  • Weekly digest email — automated Monday summary
  • Status change notifications
Exit criteria: TLs can capture and evaluate use cases independently. Platform sends meaningful emails that drive return visits.
D
Weeks 10–12

Sprints, Relations & Learning

Complete the sprint workflow, relations view, and the TL learning path.

  • Sprint brief generator — guided flow for Spikes/Sprints/Specials
  • Sprint activity type — special cards with own status flow
  • Relations view — React Flow dependency graph
  • Stream view — personalized activity feed
  • TL MiniMBA — learning path, initial content modules, progress tracking
Exit criteria: TLs can request sprints from the platform, all four playbook tabs are functional, and first MiniMBA modules are available.
Existing Clients

Transition Approach

The shift from execution support to Transformation OS needs careful navigation with current clients.

How We Roll It Out

  • Deploy improvements incrementally — clients see a better product over the quarter
  • Pilot quarterly refresh with one client first (likely Acito) before broader rollout
  • Biweekly check-in cadence tapers as platform proves it can carry the load
  • Frame it as "investing in making your transformation more self-sufficient"

Grandfathering

  • Existing clients keep current engagement terms through contract period
  • New features presented as automatic platform upgrades
  • Pricing shift (setup fee + subscription) applies to new engagements or renewals only

To Discuss

Open Questions

Decisions to make before or during implementation. Each has a recommendation.

1

Rich text format for narratives

Markdown (simple, renders everywhere) or structured blocks (more flexible, harder to build)?

Rec: Start with Markdown.
2

Quarterly refresh scheduling

Who sets the date? Auto-calculate from engagement start, or manually set by consultant?

Rec: Manually set by consultant, with option to auto-generate.
3

Comment permissions

Can any client user comment, or only the transformation lead?

Rec: Any authenticated user in the org. Keep it open.
4

Sprint brief approval flow

Does the consultant need to approve before a sprint is "real"?

Rec: Yes — draft → submitted → approved by consultant.
5

MiniMBA content creation

Who writes the modules? How many for launch?

Rec: Co-founders author initial 10-15 modules. AI-assisted drafting from existing Tatoma IP.
6

Use case library sharing

Should use cases be shareable across clients (anonymized)?

Rec: Not in Q2. Per-client first, cross-pollination later.
7

Notification volume

How many emails per week is too many?

Rec: Cap at weekly digest + activity-triggered alerts. Let users toggle off.
